## 1. Executive Summary

Surdej v1.0 is a **generic starter framework** — a forkable monorepo that provides
authentication, skinning, AI chat, worker orchestration, knowledge management, and a
convention-based extensibility model. No domain-specific business logic ships in the core.

The plan is divided into **9 phases (0–8)** spanning approximately **18 weeks**. Each phase
has explicit deliverables, maps to specific spec documents, and builds on the previous phase.

Two domain examples (PDF Refinery PDF Refinery and Nexi LAKA Dispatch) are built in Phase 7 to
prove the extensibility model works end-to-end.

## 2. Guiding Principles

| # | Principle | Source |
|---|-----------|--------|
| 1 | **Generic over specific** — no domain logic in core | Spec 00 |
| 2 | **Command-driven navigation** — every page is a command, never `<Link>` | Spec 02, CLAUDE.md |
| 3 | **Convention over import** — domains extend via manifests, not imports | Spec 16 |
| 4 | **API is the sole integration point** — frontend ↔ API only | Spec 01, 02, 03 |
| 5 | **Segmented Prisma schemas** — each worker/domain owns its data | Spec 01 |
| 6 | **Feature flags gate everything** — new features start at Ring 1 | Spec 09 |
| 7 | **Disposable lifecycle** — every resource cleanup follows IDisposable | Spec 13 |
| 8 | **TypeScript only for workers** — unified toolchain, Prisma for all data access | Spec 01, 15 |

### Topology Data Model

A **Topology** is a declarative graph that describes infrastructure, code structure, or data flow. The data is **generated** (by CLI scanners) or **hand-authored**, and the core platform provides a **Google Maps-like viewer** with two view modes. Derived projects provide the data; the platform provides the viewer.

```typescript
// contracts/topology.d.ts

/** Top-level definition — one per topology ("architecture", "codebase", etc.) */
export interface TopologyDefinition {
  id: string;                          // e.g. "pdf-refinery-architecture", "nexi-codebase"
  type: 'infrastructure' | 'codebase' | 'data-flow' | 'custom';
  name: string;                        // "Infrastructure Architecture"
  description?: string;
  icon?: string;                       // Lucide icon name
  commandId: string;                   // Auto-registered navigable command
  generatedAt?: string;                // ISO timestamp — set by CLI generator
  generatedBy?: string;                // "cli:topology:infra" or "manual"
  layers: TopologyLayer[];             // Toggleable layers (like Google Maps layers)
  actors?: TopologyActor[];            // External entities (Developer, End User, Bot)
  connections: TopologyConnection[];   // Edges between nodes
}

/** A layer groups related nodes — toggleable in the viewer (like map layers) */
export interface TopologyLayer {
  id: string;                          // e.g. "dev", "prod", "packages", "workers"
  label: string;
  icon?: string;
  visible: boolean;                    // Default visibility
  nodes: TopologyNode[];               // Top-level nodes in this layer
}

/** Recursive node — children nest infinitely */
export interface TopologyNode {
  id: string;
  label: string;
  icon?: string;                       // Lucide icon name
  description: string;
  color: string;                       // Tailwind gradient classes
  level: number;                       // Nesting depth (0 = top-level)
  minZoomToShow?: number;              // Only render in map view above this zoom
  children?: TopologyNode[];           // Nested sub-nodes
  // Links & metadata
  urls?: { label: string; href: string }[];
  properties?: TopologyProperty[];     // Shown in the property pane
  tags?: string[];                     // Searchable/filterable labels
  // Layout hints (optional — auto-layout fills these if missing)
  position?: { x: number; y: number };
  size?: { width: number; height: number };
}

/** A key-value property displayed in the property pane */
export interface TopologyProperty {
  key: string;                         // e.g. "Runtime", "Port", "Language"
  value: string;                       // e.g. "Node.js 22", "5001", "TypeScript"
  icon?: string;
  copyable?: boolean;                  // Show copy button
  href?: string;                       // Make value a link
}

export interface TopologyActor {
  id: string;
  label: string;
  icon?: string;
  description: string;
  color: string;
  properties?: TopologyProperty[];
}

export interface TopologyConnection {
  from: string;                        // Node ID
  to: string;                          // Node ID
  label?: string;                      // Edge label (e.g. "HTTPS", "NATS", "CI/CD")
  animated?: boolean;                  // Animated dash line
  style?: 'solid' | 'dashed' | 'dotted';
}
```

**Key design decisions:**

| Decision | Rationale |
|----------|----------|
| **Layers** instead of environments | More flexible — you can toggle "Docker Services", "Azure", "Kubernetes" independently, just like map layers |
| **`properties[]`** on nodes | Powers the property pane — structured key-value pairs that the viewer renders without domain-specific UI code |
| **`position?` is optional** | Nodes can omit position — the auto-layout engine (dagre/ELK) calculates positions. Hand-authored topologies can pin positions. |
| **`generatedAt` + `generatedBy`** | Topologies are often generated by CLI scanners. These fields track freshness and provenance. |
| **`tags[]`** on nodes | Enables search/filter in both map view and explorer view |
| **`type` discriminator** | The viewer can apply type-specific defaults (e.g. colour palettes for infrastructure vs codebase) |

### Topology Generators (CLI)

Topology data is generated automatically by CLI commands that scan the project:

| Generator | Command | Scans | Output |
|-----------|---------|-------|--------|
| **Infrastructure** | `pnpm cli:topology:infra` | `docker-compose.yml`, `infra/k8s/*.yaml`, `.github/workflows/`, Cloudflare config, `prisma/schema/` | `TopologyDefinition` with layers: Dev (Docker Compose), Prod (AKS + Azure + Cloudflare), CI/CD (GitHub Actions) |
| **Codebase** | `pnpm cli:topology:code` | `pnpm-workspace.yaml`, `package.json` (all workspaces), `tsconfig.json` references, `prisma/schema/*.prisma`, route files (`routes/**/page.tsx`) | `TopologyDefinition` with layers: Packages, Apps, Workers, Routes, Prisma schemas |
| **Data flow** | `pnpm cli:topology:flow` | NATS subjects in code, API route definitions, worker job handlers | `TopologyDefinition` with layers: API endpoints, NATS subjects, Worker job handlers |

Generators write to `domains/<name>/topologies/<type>.generated.ts`. The `.generated.ts` suffix indicates auto-generated files. Hand-authored topologies use plain `.ts`.

### Skin Data Model (Core Schema)

```prisma
model Skin {
  id          String   @id @default(uuid())
  name        String
  description String?
  isBuiltIn   Boolean  @default(false)   // true = seeded, cannot be deleted
  createdBy   String?                     // userId of creator (null for built-in)
  branding    Json                        // { appName, logo, primaryColor, fontFamily }
  sidebar     Json                        // ordered list of { commandId, group }
  theme       Json?                       // { defaultMode: "light" | "dark" }
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  preferences UserSkinPreference[]
  @@schema("public")
}

model UserSkinPreference {
  id           String   @id @default(uuid())
  userId       String
  skinId       String
  isDefault    Boolean  @default(false)   // user's default skin on login
  isActive     Boolean  @default(false)   // currently active skin this session
  customOrder  Json?                      // user-level sidebar order overrides
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt

  user         User     @relation(fields: [userId], references: [id])
  skin         Skin     @relation(fields: [skinId], references: [id])

  @@unique([userId, skinId])
  @@schema("public")
}
```

**Key design decisions:**
- `Skin` is a shared record — multiple users can use the same skin.
- `UserSkinPreference` is the join table — tracks which skin a user has active and which is their default.
- `isDefault` marks the skin the user sees on login (one per user, enforced in service layer).
- `isActive` marks the user's current session skin (allows switching without changing default).
- `customOrder` allows a user to reorder sidebar items within a skin without creating a full clone.
- Built-in skins (`isBuiltIn: true`) are seeded by the platform and cannot be deleted.

### Skin API Endpoints

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/skins` | GET | List all skins (built-in + custom) |
| `/api/skins` | POST | Create a custom skin (sets `createdBy` to current user) |
| `/api/skins/:id` | GET | Get skin details |
| `/api/skins/:id` | PUT | Update a skin (owner or admin only; built-in skins are immutable) |
| `/api/skins/:id` | DELETE | Delete a custom skin (owner or admin only; built-in skins cannot be deleted) |
| `/api/skins/:id/clone` | POST | Clone a skin (creates a new custom skin from an existing one) |
| `/api/skins/me` | GET | Get current user's active skin (falls back to default, then platform default) |
| `/api/skins/me` | PUT | Set the active skin for the current session |
| `/api/skins/me/default` | PUT | Set the user's default skin (applied on login) |
| `/api/skins/me/order` | PUT | Save custom sidebar order overrides for the active skin |

## Multi-Tenancy Strategy

Surdej v1.0 uses a **shared database with `tenantId` column** isolation strategy. This balances simplicity (one DB, one connection pool) with proper data separation. All tenant-scoped models carry an optional `tenantId` FK; global resources (e.g. FeatureFlags) remain tenant-agnostic.

### Tenant Model

```
model Tenant {
  id          String   @id @default(uuid())
  name        String                          // Display name, e.g. "Demo - Burger Restaurant"
  slug        String   @unique                // URL-safe identifier, e.g. "demo-burger"
  description String?
  logoUrl     String?
  metadata    Json?                           // { industry, locale, timezone, brandColor, ... }
  isDemo      Boolean  @default(false)
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
}
```

### Tenant-Scoped Models

| Model | `tenantId` | Notes |
|-------|-----------|-------|
| `User` | ✅ Optional | Users can belong to a tenant; platform admins may be tenant-less |
| `Skin` | ✅ Optional | Built-in skins are global; custom skins belong to a tenant |
| `Article` | ✅ Optional | Knowledge articles are tenant-scoped |
| `Template` | ✅ Optional | Templates can be global (built-in) or tenant-specific |
| `TrainingModule` | ✅ Optional | Training content is per-tenant |
| `AiConversation` | ✅ Optional | Chat history is tenant-isolated |
| `FeedbackEntry` | ✅ Optional | Feedback is tenant-scoped for privacy |
| `WorkerRegistration` | ✅ Optional | Workers can be shared or tenant-dedicated |
| `Blob` | ✅ Optional | Uploaded files are tenant-scoped |
| `McpServerConfig` | ✅ Optional | MCP servers can be tenant-specific |
| `FeatureFlag` | ❌ Global | Feature flags are platform-wide, not per-tenant |
| `Session` | ❌ Inherits via User | Session belongs to a user who belongs to a tenant |
| `UserSkinPreference` | ❌ Inherits via User + Skin | Cross-reference — tenant resolved via parent |
| `LearnerProgress` | ❌ Inherits via User + TrainingModule | Cross-reference |

### Tenant Resolution (API)

The API resolves the active tenant on each request using this priority chain:

1. **JWT claim** — `tenantId` embedded in the access token (enterprise mode via Entra ID custom claims)
2. **Request header** — `X-Tenant-Id` header (used by CLI and workers)
3. **User's default tenant** — from `User.tenantId` in the database
4. **Demo tenant** — Falls back to the demo tenant in demo mode

```typescript
// Fastify middleware sketch
async function resolveTenant(request: FastifyRequest) {
    const tenantId =
        request.user?.tenantId                          // JWT claim
        ?? request.headers['x-tenant-id']               // Explicit header
        ?? (await getUserDefaultTenant(request.user.id)) // DB lookup
        ?? getDemoTenantId();                            // Fallback
    request.tenantId = tenantId;
}
```

### Query Scoping (API)

Every tenant-scoped query includes a `WHERE tenantId = ?` clause. This is enforced via a Prisma extension:

```typescript
// prisma-tenant-extension.ts
export function withTenantScope(prisma: PrismaClient, tenantId: string) {
    return prisma.$extends({
        query: {
            $allModels: {
                async findMany({ args, query }) {
                    args.where = { ...args.where, tenantId };
                    return query(args);
                },
                async create({ args, query }) {
                    args.data = { ...args.data, tenantId };
                    return query(args);
                },
                // ... findFirst, update, delete similarly
            },
        },
    });
}
```

A `TenantContext` provider mirrors the `SkinContext` pattern:

| Feature | Implementation |
|---------|---------------|
| **Active tenant** | Loaded from `GET /tenants/me` on mount |
| **All tenants** | Loaded from `GET /tenants` — dropdown options |

### Worker Tenant Awareness

Workers receive `tenantId` in job payloads. The `WorkerBase` class:

1. Extracts `tenantId` from the NATS message payload
2. Creates a tenant-scoped Prisma client for the job
3. Includes `tenantId` in all log entries
4. Respects `affinityKey` for tenant-local job routing

### Demo Seeding

The seed script creates a demo tenant **"Demo - Burger Restaurant"** (`slug: demo-burger`) with:
- Industry metadata (food-service, en-US locale)
- 4 demo users associated with the tenant
- All knowledge content (articles, templates, training modules) belongs to this tenant
- Demo-mode login automatically selects this tenant

### Migration Path

To add a new tenant after v1.0:

1. Create the `Tenant` record via API or seed script
2. Invite users (or create demo users) with `tenantId` set
3. Optionally create tenant-specific skins and templates
4. Workers automatically scope to the tenant via job payloads

| # | Risk | Likelihood | Impact | Mitigation |

| R1 | Vite plugin for domain discovery is fragile | Medium | High | Build with comprehensive error reporting. Fall back to manual `index.ts` if plugin fails. |
