# 16 — Extensibility Model

## Problem

Traditional monorepos force derived projects into one of two patterns, both with friction:

1. **Fork and diverge** — fork the monorepo, modify everything inline. Merge conflicts
   on every upstream sync. Works early, breaks later.
2. **Import shared packages** — `npm install @surdej/core`. A central body must control
   what goes into the package. Versioning, publishing, breaking changes, approval queues.
   Derived projects can't move faster than the core team releases.

Surdej needs a third path: **extend by convention, not by import.**

---

## Design Principle: Convention Over Import

Domain extensions plug into the platform through **well-known file locations**, **JSON/TS
manifests**, and **runtime discovery** — not by importing from a shared package.

The platform defines **contracts** (shapes of manifests, naming conventions, folder structures).
Extensions satisfy those contracts. The platform discovers and loads them at build time or
runtime. No `import { CommandDefinition } from "@surdej/core"` required.

---

## How It Works

### 1. Manifest-Based Discovery (Frontend)

The frontend discovers domain modules at **build time** via file-system conventions:

```
apps/frontend/src/domains/
├── my-domain/
│   ├── manifest.ts          ← discovered automatically
│   ├── pages/
│   ├── components/
│   └── ...
└── another-domain/
    ├── manifest.ts          ← discovered automatically
    └── ...
```

A Vite plugin scans `src/domains/*/manifest.ts` at build time and generates a registry:

```typescript
// auto-generated by vite-plugin-surdej-domains at build time
// DO NOT EDIT — regenerated on every build / HMR
import { manifest as myDomain } from "@/domains/my-domain/manifest";
import { manifest as anotherDomain } from "@/domains/another-domain/manifest";

export const domainManifests = [myDomain, anotherDomain];
```

No central `domains/index.ts` to maintain. Drop a folder with a `manifest.ts`, restart dev
server, done.

### 2. Manifest Shape (The Contract)

The manifest is a plain TypeScript object literal. The domain does **not** import types from
`@surdej/core` — it just matches the expected shape:

```typescript
// domains/my-domain/manifest.ts
// No imports from core. Just satisfy the contract shape.

export const manifest = {
  id: "my-domain",
  name: "My Domain",

  // Commands this domain provides
  commands: [
    {
      id: "domain.my-domain.dashboard",
      label: "My Domain Dashboard",
      icon: "Briefcase",
      keywords: ["my-domain", "dashboard"],
      category: "domain",
      target: { type: "route", path: "/domains/my-domain" },
    },
  ],

  // Routes this domain mounts
  routes: [
    {
      path: "/domains/my-domain",
      lazy: () => import("./pages/DashboardPage"),
    },
    {
      path: "/domains/my-domain/:id",
      lazy: () => import("./pages/DetailPage"),
    },
  ],

  // Context keys this domain contributes
  contextKeys: [
    { key: "domain.my-domain.hasData", type: "boolean", default: false },
  ],

  // Sidebar items (only shown when the skin includes this domain)
  sidebarItems: [
    { commandId: "domain.my-domain.dashboard", group: "domain" },
  ],
} as const;
```

The `as const` assertion provides full type inference without any import. TypeScript infers
the exact shape. The platform validates at build time using its own type definitions — the
domain never needs to know about them.

### 3. Type Safety Without Importing

The platform defines the contract types internally. The Vite plugin validates each manifest
against the contract at build time and surfaces errors:

```
[surdej] ✖ domains/broken-domain/manifest.ts: Missing required field "id"
[surdej] ✖ domains/broken-domain/manifest.ts: commands[0].target.type must be "route" | "href" | "virtual-page" | "action"
```

For domains that **want** type safety during development (optional), they can use a
lightweight type-only package:

```typescript
// Optional: import types only (zero runtime, zero dependency)
import type { DomainManifest } from "@surdej/types";

export const manifest: DomainManifest = { ... };
```

`@surdej/types` is a **types-only** package — no runtime code, no functions, no classes.
It's never bundled and never creates a dependency chain. It's purely developer ergonomics.
Domains can function without it.

### 4. API Extension (Fastify Plugins)

API domain modules follow the same pattern — discovered by convention:

```
apps/api/src/domains/
├── my-domain/
│   ├── plugin.ts            ← discovered automatically
│   ├── routes.ts
│   ├── service.ts
│   ├── schema.ts            ← Zod schemas
│   └── types.ts
```

The API server scans `src/domains/*/plugin.ts` at startup and registers each as a
Fastify plugin:

```typescript
// domains/my-domain/plugin.ts
import type { FastifyInstance } from "fastify";

export default async function (app: FastifyInstance) {
  const { myDomainRoutes } = await import("./routes");
  app.register(myDomainRoutes, { prefix: "/api/domains/my-domain" });
}

// Metadata for the plugin registry (optional, for MCP tool exposure)
export const meta = {
  id: "my-domain",
  name: "My Domain API",
  description: "Domain-specific API endpoints",
};
```

No central `domains/index.ts`. The server discovers and loads plugins automatically.

### 5. Worker Extension (Standalone Projects)

Workers are **fully standalone** TypeScript projects. They communicate with the platform
exclusively through NATS subjects and their own Prisma schema. No imports from `@surdej/core`
at all — just NATS messages and database writes.

```
workers/my-worker/
├── package.json             ← standalone, no @surdej/* dependencies
├── tsconfig.json
├── Dockerfile
├── prisma/
│   └── schema/
│       └── my_worker.prisma ← owns its own schema segment
└── src/
    ├── index.ts             ← NATS connection, heartbeat, job handlers
    ├── handlers/
    │   └── process-thing.ts
    └── types.ts             ← local types, no imports from core
```

Workers discover the platform through environment variables:

```typescript
// workers/my-worker/src/index.ts
const NATS_URL = process.env.NATS_URL ?? "nats://localhost:4224";
const WORKER_TYPE = "my-worker";

// Connect to NATS
const nc = await connect({ servers: NATS_URL });

// Register with worker registry (just publish a message — no import needed)
await nc.publish("worker.register", encode({
  workerId: crypto.randomUUID(),
  workerType: WORKER_TYPE,
  version: "1.0.0",
  capabilities: ["process-thing"],
  prismaSchema: "my_worker",
  maxConcurrency: 5,
}));

// Subscribe to jobs
const sub = nc.subscribe(`job.${WORKER_TYPE}.*`);
for await (const msg of sub) {
  const action = msg.subject.split(".").pop();
  const result = await handlers[action](decode(msg.data));
  msg.respond(encode(result));
}
```

**Zero coupling to the monorepo.** A worker can be developed in a completely separate git
repository. It just needs to know the NATS subjects and message shapes.

### 6. Skin Extension

Skins follow the same manifest pattern:

```typescript
// skins/my-brand/manifest.ts
export const manifest = {
  id: "my-brand",
  name: "My Brand",
  branding: {
    appName: "My App",
    logo: "/assets/my-brand/logo.svg",
    primaryColor: "#A6192E",
    fontFamily: "Montserrat",
  },
  sidebar: [
    { commandId: "navigate.home", group: "main" },
    { commandId: "navigate.chat", group: "main" },
    { commandId: "domain.my-domain.dashboard", group: "domain" },
    { commandId: "navigate.settings", group: "footer" },
  ],
  theme: { defaultMode: "light" },
} as const;
```

## Development Modes

The extensibility model supports three development modes:

### Mode 1: In-Monorepo Development

Domain code lives inside the Surdej monorepo in `domains/`, `workers/`, `skins/`.
Simplest setup. Best for small teams and early-stage projects.

```
surdej-v1/
├── apps/frontend/src/domains/my-domain/   ← domain code here
├── apps/api/src/domains/my-domain/        ← API extension here
├── workers/my-worker/                      ← worker here
└── skins/my-brand/                         ← skin here
```

### Mode 2: Linked External Projects

Domain code lives in **separate git repositories**, linked into the monorepo via:

- **pnpm workspace protocol** — `"my-domain": "workspace:*"` in `pnpm-workspace.yaml`
- **Symlinks** — `ln -s ../my-domain-repo/frontend apps/frontend/src/domains/my-domain`
- **Git submodules** — for teams that prefer them

The Vite plugin and API scanner don't care where the files come from — they just
scan the `domains/` directory.

### Mode 3: Fully External Workers

Workers can be developed in **completely separate repositories** with no code-level
dependency on Surdej. They only need:

1. NATS connection URL
2. Knowledge of NATS subject conventions (`job.<type>.<action>`, `worker.register`, etc.)
3. PostgreSQL connection URL (for their schema segment)
4. Message payload shapes (documented as JSON schemas)

This mode is ideal for teams that want to develop independently without even cloning
the Surdej repo.

## Contract Documentation

Instead of `@surdej/core` as the source of truth for types, contracts are documented as
**JSON schemas** and **TypeScript type files** in a dedicated location:

```
contracts/
├── domain-manifest.schema.json     ← JSON Schema for DomainManifest
├── domain-manifest.d.ts            ← TypeScript type declarations
├── worker-messages.schema.json     ← NATS message payload schemas
├── worker-messages.d.ts
├── skin-manifest.schema.json
├── skin-manifest.d.ts
├── api-plugin.schema.json
├── api-plugin.d.ts
└── README.md                       ← Human-readable contract documentation
```

These contracts are:
- **Versioned** alongside the monorepo (same git repo).
- **Published as a types-only package** (`@surdej/types`) for optional dev ergonomics.
- **Published as JSON schemas** for external tools and non-TypeScript consumers.
- **Never contain runtime code** — purely shapes and validation.

---

## What `@surdej/core` Becomes

`@surdej/core` still exists but its role changes. It's an **internal** package used only
by the platform's own code (`apps/api`, `apps/frontend/src/core/`). Domains and workers
do **not** import from it.

| Consumer | Uses `@surdej/core`? | Uses `@surdej/types`? |
|----------|---------------------|----------------------|
| `apps/api/src/core/` | ✅ Yes | ✅ Yes |
| `apps/frontend/src/core/` | ✅ Yes | ✅ Yes |
| `apps/frontend/src/domains/*` | ❌ No | 📎 Optional (types only) |
| `apps/api/src/domains/*` | ❌ No | 📎 Optional (types only) |
| `workers/*` | ❌ No | 📎 Optional (types only) |
| `skins/*` | ❌ No | 📎 Optional (types only) |
| External repos | ❌ No | 📎 Optional (types only) |

---

## Build-Time vs Runtime Discovery

| Extension Point | Discovery | Mechanism |
|----------------|-----------|-----------|
| Frontend domains | Build time | Vite plugin scans `src/domains/*/manifest.ts` |
| Skins | Build time | Vite plugin scans `src/skins/*/manifest.ts` |
| API domains | Startup (runtime) | Fastify scans `src/domains/*/plugin.ts` |
| Workers | Runtime | NATS `worker.register` message |
| MCP tools | Runtime | Worker registration includes tool definitions |

---

## The Anti-Patterns This Avoids

| Anti-Pattern | How We Avoid It |
|-------------|-----------------|
| Central package bottleneck | Domains don't import `@surdej/core`. Contracts are shapes, not code. |
| `domains/index.ts` barrel file | Auto-discovery via file-system scanning. No manual registration. |
| Breaking changes in shared types | Contracts are versioned JSON schemas. Old shapes stay supported. |
| "Ask permission to extend" | Drop a folder, satisfy the contract, restart. No PR to core needed. |
| Tight coupling to monorepo | Workers are standalone. Domains can be symlinked from external repos. |
| Single team controls the core | Core team controls the *platform*. Domain teams control their *domains*. |
